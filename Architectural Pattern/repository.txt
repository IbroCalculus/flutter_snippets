
=================== WHAT IS A REPOSITORY? ===================
A Repository is about deciding where data should come from and providing a clean interface to the rest of the app.

--------- Core Idea (Very Important) ----------
Instead of your app saying: "I need data from the API". It should say: "I need books".

The repository decides: API?, Local database?, Cache?, Offline storage? The caller doesn't care.


------------- Simple Definition --------
A Repository is a middle layer between Services and the rest of the app.
It answers: "Where should I get the data from?"

--------------- Architecture Flow ---------
        Screen / Provider
            ↓
        Repository
            ↓
        Service
            ↓
           API


--------------- Why We Don't Skip Repository -------------
Without repository: Provider → Service → API

Problem:
    UI logic gets tightly coupled to data sources
    Hard to switch API → local DB
    Hard to test

With repository: Provider → Repository → Service → API
    Now UI never knows where data comes from.


----------------- Code Example (Conceptual) ---------------

abstract class BookRepository {
  Future<List<BookModel>> getBooks();
}

This says: "Any repository must be able to provide books."


---------------- Repository Implementation ---------------

class BookRepositoryImpl implements BookRepository {
  final BookService service;

  BookRepositoryImpl(this.service);

  @override
  Future<List<BookModel>> getBooks() {
    return service.getBooks();
  }
}
